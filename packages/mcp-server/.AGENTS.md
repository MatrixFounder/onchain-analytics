# Directory: packages/mcp-server/

## Purpose

The one workspace package created in M0: the `@onchain-intel/mcp-server` package — a stdio-only
MCP server skeleton (D3/D5, ADR-001) with an env module (zod, all keys optional in M0, R-12) and
one dummy tool, `onchain_ping` (R-10), built strictly per ARCHITECTURE.md §3.2/§4.1/§5.2 (task
001-2). `test/` (unit + stdio E2E, task 001-3) makes the M0 exit-criterion — "`onchain_ping`
answers over stdio" — verifiable in CI (R-6) instead of only by manual smoke test.

**Task 003-6 (R-23/R-24)** extended `src/env.ts`'s `EnvSchema` with 4 M1-optional keys
(`COINGECKO_API_KEY`, `DUNE_API_KEY`, `ONCHAIN_PG_URL`, `DATA_DIR` — still no required key,
`EnvSchema.parse({})` keeps succeeding) and added the first-ever import of `@onchain-intel/core`
into this package (`test/env-degradation.integration.test.ts`, exercising `loadEnv()` together with
`@onchain-intel/core`'s `CapabilityRegistry`/`createPgHistoryAdapter` to prove the missing-key
degradation contract end-to-end). No MCP tool/handler wiring yet at that point — `createServer`,
`server.ts`, and `src/tools/*.ts` were unchanged.

**Task 003-7 (R-16..R-20, closes TASK-003's M1 tool scope)** finishes the wiring 003-6 deferred:
4 new provider-backed MCP tools (`src/tools/get-token.ts`/`wallet-balances.ts`/`new-pairs.ts`/
`protocol-tvl.ts`, plus a shared `resolve-capability.ts` helper), `server.ts`'s `createServer` now
takes an injectable `registry?: CapabilityRegistry`, and `index.ts` builds the ONE real,
network-capable `CapabilityRegistry` (all 9 real `@onchain-intel/core` adapters + the real
two-level cache) exactly once at startup. `onchain_ping`/`ping.ts` are unchanged (R-20). See the
per-file sections below for the full design.

## Files

### src/env.ts

`EnvSchema` (zod) — every key **optional** (R-12 in M0, R-23 for the 4 new M1 keys added this
task; `EnvSchema.parse({})` has never thrown and still doesn't). Deliberately not `.strict()`: the
real input is `process.env`, which carries hundreds of unrelated keys — zod's default object
behavior strips them instead of rejecting them.

`LOG_LEVEL?: 'debug'|'info'|'warn'|'error'` (reserved, unused) — M0.

**Task 003-6 (R-23) added 4 M1 keys, all optional, ARCHITECTURE.md §3.2/§10.3:**
`COINGECKO_API_KEY?: string`/`DUNE_API_KEY?: string` (read by `@onchain-intel/core`'s
`coingecko`/`dune` adapters — this schema only validates presence/shape, never reads or forwards
the value itself), `ONCHAIN_PG_URL?: string` (the `pg-history` read-only Postgres DSN, validated
with `z.string().url()`), `DATA_DIR?: string` (the cache directory override
`@onchain-intel/core`'s `resolveDataDir()` already reads, task 003-3). None of the 4 is required —
an empty `.env` remains a fully valid, working configuration (UC-1/R-27); each capability that
needs one of them degrades explicitly through `@onchain-intel/core`'s own `isAvailable()` contract
(003-2/003-5) rather than this schema rejecting the empty env.

**`ONCHAIN_PG_URL` zod `.url()` — empirically verified, not guessed (ARCHITECTURE.md §11 dev-time
check):** zod 4.4.3's WHATWG URL parsing accepts a realistic `postgres://user:pass@host:5432/db`
connection string, including a percent-encoded special character in the password
(`postgres://user:p%40ss@host:5432/db`) and a trailing query string
(`?sslmode=require`) — verified via a throwaway `tsx -e` one-liner (this task's own acceptance
command) before committing to `.url()` over a hand-rolled prefix/regex check. No fallback was
needed.

**Empty-string idiom, generalized (task 003-6):** the private `emptyAsUndefined<T>(schema)` helper
wraps `z.preprocess((v) => (v === '' ? undefined : v), schema)` — the exact fix task 001-4 applied
to `LOG_LEVEL` alone, now shared by all 5 keys (implementation choice, not an architectural one:
avoids repeating the same preprocess lambda 5 times; `LOG_LEVEL`'s own behavior is unchanged, only
its declaration is now expressed via the shared helper). A shell exporting `KEY=` (empty string, not
absent) behaves identically to the var being unset entirely, for every key in this schema.

**Stub-first evidence (task 003-6):** Phase 1 declared the 4 keys as bare `z.string().optional()`
(`ONCHAIN_PG_URL` as `z.string().url().optional()`), with NO `emptyAsUndefined` wrapper yet — this
made the file typecheck clean (`tsc --noEmit`, the Phase 1 gate) while two new
`test/env.test.ts` cases (asserting `KEY: ''` parses to `undefined`) genuinely failed red (captured
before the fix: `ONCHAIN_PG_URL: ''` threw a `ZodError` since `''` isn't a valid URL; the 3 plain
string keys silently kept `''` instead of `undefined`). Phase 2 added the `emptyAsUndefined` wrapper
to all 5 keys, turning both red cases green with no other changes.

`loadEnv(raw?: NodeJS.ProcessEnv): Env` — when `raw` is omitted (the real `index.ts` call site),
first tries `process.loadEnvFile()` (Node's built-in `.env` loader, stable ≥20.12 — no `dotenv`
dependency; a missing `.env` is expected in M0 and silently ignored, `ENOENT` is swallowed via a
local `isEnoent()` guard; any other load error is logged to stderr but does not abort). Then
`EnvSchema.safeParse(...)`: on failure, writes `onchain-intel-mcp-server: invalid environment
configuration for: <KEY, KEY, ...>` to **stderr** (key names only, collected from
`error.issues[].path` — **never the value**, D10) and throws; `loadEnv` itself never calls
`process.exit`, so it stays a plain, unit-testable function — `index.ts` decides how to turn the
throw into a process exit.

### src/tools/ping.ts

`PingInputSchema = z.object({}).strict()` / `PingOutputSchema = z.object({ ok: z.literal(true),
service: z.literal('onchain-intel-mcp-server'), version: z.string(), ts: z.number().int() })` —
the single source of truth for both runtime validation and the MCP tool-schema (no hand-written
JSON-Schema duplicate).

`pingHandler(input, ctx: { version })` — pure function, unit-testable without a transport; returns
`PingOutputSchema.parse({ ok: true, service: 'onchain-intel-mcp-server', version: ctx.version, ts:
Date.now() })`.

`registerPingTool(server, ctx: { version })` — calls `server.registerTool('onchain_ping', {
description, inputSchema: PingInputSchema, outputSchema: PingOutputSchema }, cb)`. `registerTool`
(not the deprecated `tool()`) was picked by reading the installed SDK's `server/mcp.d.ts`
directly (vendor-drift guard, ARCHITECTURE §11) — it accepts a **full zod schema** as
`inputSchema`/`outputSchema` (its `AnySchema` union includes `z4.$ZodType`, which `z.object(...)`
satisfies), not just a raw shape, so `PingInputSchema`/`PingOutputSchema` are passed straight
through, unmodified. The callback returns both `content` (a JSON-stringified text block) and
`structuredContent` (the parsed object) — the SDK requires `structuredContent` whenever
`outputSchema` is set (verified by reading `mcp.js`'s `validateToolOutput`).

### src/tools/resolve-capability.ts (new, task 003-7)

Shared `registry.resolve()` wrapper for the 4 new M1 tools — extracted because all 4 handlers need
the byte-identical try/catch + `_meta.cache`-shape logic (DRY, developer-guidelines §1.6 "internal
abstraction — apply professional judgment"; mirrors `@onchain-intel/core`'s own precedent of
sharing small helpers across sibling adapter files, e.g. `not-implemented-error.ts`/
`dash-metrics.ts`). `onchain_ping` does NOT use this (R-20, unchanged).

`CacheMeta { status: 'hit'|'miss'; ageMs?: number; provider: string; capability: string }` — the
`_meta.cache` shape every one of the 4 new tools attaches to its MCP response (ARCHITECTURE.md
§3.2/§5.1, R-15), deliberately a sibling of `structuredContent`/`content`, never folded into
either. `ResolveOutcome = {ok: true; output: unknown; cache: CacheMeta} | {ok: false; reason:
string}`. `resolveCapability(registry, capability, chain, args)` calls `registry.resolve(...)`,
never throws: any rejection (in practice always `CapabilityUnavailableError`, ARCHITECTURE.md
§9.1) becomes `{ok: false, reason: error.message}` — that error's own constructor already builds a
message with no secret values (D10), forwarded verbatim.

### src/tools/get-token.ts, wallet-balances.ts, new-pairs.ts, protocol-tvl.ts (new, task 003-7, R-16..R-19)

Same pattern as `ping.ts`: zod in/out schema (single source of truth for both runtime validation
and the MCP tool-schema) + a pure handler (async here, since it calls `registry.resolve()`; still
unit-testable without a transport, `test/tools/*.test.ts`) + a `registerXTool(server, {registry})`
wiring function. **All 4 input schemas narrow `chain` to `z.enum(['ethereum', 'solana'])`** — NOT
the full `ChainSchema` (task 003-7 reviewer note, Major-2 from architecture review cycle 1):
`isValidAddress`/`normalizeAddress` don't implement Dash address validation (`dash-platform` works
through `Snapshot`, not `Token`/`Wallet`), so accepting `'dash'` here would be a value that always,
unconditionally fails validation — a misleading contract. `new-pairs.ts`/`protocol-tvl.ts` each
declare this enum ONCE as a local `SUPPORTED_CHAIN` constant, reused for both their input's and
output's `chain` field (keeps the task's own acceptance grep — "narrowed in all 4 tool files" — at
exactly 4 matches across `src/tools/`, one per file, instead of also matching output-schema/
docstring repeats).

- **`get-token.ts`** (R-16) — `GetTokenInputSchema` (`chain` + `address: z.string().min(1)`,
  `.strict()`, `superRefine` → `isValidAddress(chain, address)`). Output is `TokenSchema`
  re-exported from `@onchain-intel/core` verbatim (task scope: "outputs = canonical types from
  `@onchain-intel/core`") — no new zod schema for a shape that already has one. Capability:
  `token.metadata` (not `token.price` — both route to `coingecko` and its `normalize()` produces
  the byte-identical `Token` either way, task 003-4's own adapter docstring; `token.metadata` is
  the more accurate name for this tool's full-object contract). `getTokenHandler` re-normalizes
  `input.address` via `normalizeAddress` before it becomes part of the cache-key `args` (the
  documented `net/args-hash.ts` contract: args must be the normalized, post-zod-validation tool
  input).
- **`wallet-balances.ts`** (R-17) — `WalletBalancesInputSchema` is the literal sample from
  ARCHITECTURE.md §5.1 (same `chain`+`address`+`superRefine` shape as `get-token.ts`). Output is
  `WalletSchema` verbatim; capability `wallet.balances.native`, backed by `rpc-evm`/`rpc-solana`
  per chain (OQ-1).
- **`new-pairs.ts`** (R-18) — `NewPairsInputSchema` (`chain` + `limit?:
z.number().int().positive().optional()`). Output is a NEW zod schema, `NewPairsOutputSchema =
{chain, pairs: Pool[], source, fetchedAt}` (copied literally from ARCHITECTURE.md §5.1) — not one
  of the six canonical `types/*` schemas (only `Pool`, the array element type, is); mirrors
  `defillama`'s own `ProtocolTvlResult` precedent (a tool-contract-level shape, not a canonical
  domain entity). Capability `pairs.new` → `dexscreener`, which returns a bare `Pool[]`; the
  handler wraps it: `source` = the answering adapter id (`outcome.cache.provider`), `fetchedAt` =
  `Date.now()` at response-build time (a documented implementation choice, developer-guidelines
  §1.5 — distinct from each individual `Pool`'s OWN `fetchedAt`, which may be older on a cache hit).
- **`protocol-tvl.ts`** (R-19) — `ProtocolTvlInputSchema` (`chain` + `protocolSlug:
z.string().min(1)`, no `superRefine` — a slug isn't an address). `ProtocolTvlOutputSchema` is the
  FIRST zod schema for `defillama`'s `ProtocolTvlResult` shape (`{protocol, chain, tvlUsd,
totalTvlUsd, source, fetchedAt}`, task 003-4's own plain-TS-interface, not zod, by that adapter's
  documented choice) — `tvlUsd`/`totalTvlUsd` are `.nonnegative()`, mirroring
  `PoolSchema.liquidityUsd`/`volume24hUsd`'s same constraint on analogous USD fields. Capability
  `protocol.tvl` → `defillama`.

**Shared `isError` wiring (all 4, task 003-7 reviewer note):** each `registerXTool`'s callback
checks `outcome.ok` explicitly — `false` → `{isError: true, content: [{type: 'text', text:
outcome.reason}]}`, built by hand, never relying on the SDK's automatic `isError` conversion (M0
`.AGENTS.md` documents that auto-conversion as covering only zod input-validation failures, not
business-logic errors). On success: `content` (JSON-stringified `structuredContent`),
`structuredContent` (the zod-parsed output), and `_meta: {cache: outcome.cache}` — verified
empirically (Phase 1 `tsc --noEmit`) that the SDK's `CallToolResultSchema._meta` is a
`z.core.$loose` object, so this extra `cache` key survives both the server's own
`structuredContent`/`_meta` round-trip and a client-side parse untouched, without widening any
tool's advertised `outputSchema`.

### src/server.ts (extended, task 003-7)

`createServer(deps: { env: Env; version: string; registry?: CapabilityRegistry }): McpServer` —
`registry` is now injectable (ARCHITECTURE.md §3.2/§5.2, F-1): registers all 5 tools (`ping` +
the 4 new M1 tools), each of the 4 new ones given `{registry}` as context. `deps.registry`'s
fallback (`new CapabilityRegistry(routes, new Map())`, real `routes` table from
`@onchain-intel/core`'s `providers.config.ts`, an EMPTY adapter `Map`) is a deliberately INERT
default — every capability degrades gracefully (`CapabilityUnavailableError`, "no adapter
registered for this id") rather than crashing, but it is NEVER the real, network-capable registry:
assembling all 9 real adapters + the real cache is `index.ts`'s single, explicit responsibility
(this task's own instruction, ARCHITECTURE.md §3.2 "строится один раз в index.ts") — `index.ts`
always constructs one and passes it in explicitly. This fallback exists purely so `createServer`
stays type-safe and harmless if a future caller ever omits `registry` outside that documented
production path (implementation choice, developer-guidelines §1.5). This injection point is the
ONLY mechanism "MCP E2E without network" (R-21) relies on — no global `fetch` mock, a different
implementation of the same public `CapabilityRegistry.resolve()` contract is injected here; it
works ONLY in-process (unreachable across a spawned child process boundary, F-1) — see
`test/e2e.inprocess.test.ts` below.

### src/index.ts (extended, task 003-7)

Bin entry (`#!/usr/bin/env node`, preserved by tsup/esbuild in `dist/index.js`, which is also
marked executable). Reads `version` **once** via `createRequire(import.meta.url)` +
`require('../package.json')` (cast to a local `PackageJson` interface — never left as `any`) —
chosen over a JSON import attribute (`with { type: 'json' }`) because that syntax is flaky under
the pinned TypeScript 6 / NodeNext combination this package uses; `createRequire` + `require(...)`
works identically under `tsx src/index.ts` (dev) and the bundled `dist/index.js` (both sit one
directory below the package root, so `../package.json` resolves the same way in both). Note for
ESLint: `@typescript-eslint/no-require-imports` (part of the shared `recommended` config) does
**not** flag this — the rule explicitly whitelists a `require()` call when `require` resolves to a
locally-declared variable (as `createRequire` produces), rather than the CJS global.

**`buildRegistry(env: Env): CapabilityRegistry` (new, task 003-7)** — assembles the ONE real,
network-capable registry for production: a `Map` with all 9 real `ProviderAdapter`s from
`@onchain-intel/core` (`createCoingeckoAdapter({env})`/`createPgHistoryAdapter({env})` — the two
that read an optional env key — plus the other 7, all keyless/no-arg), the real `routes` table,
and the real cache (`createCacheStore()`, no explicit `dbPath` — its `resolveDataDir()` default
already reads `process.env.DATA_DIR`, which `loadEnv()` (called just before this) has already
synced into `process.env` via `process.loadEnvFile()`). This is the single place the real
9-adapter set is ever constructed (task's own instruction) — `server.ts`'s own `registry` default
is a separate, deliberately inert fallback (see its section above).

`main()`: `loadEnv()` (its own try/catch — on throw, `process.exit(1)` immediately, no extra
logging, since `loadEnv` already wrote the stderr diagnostic) → `buildRegistry(env)` →
`createServer({ env, version, registry })` → `await server.connect(new StdioServerTransport())` —
the **only** place a transport is chosen (R-9); a future (M6) alternative HTTP-based transport
would be attached here too, `createServer` stays unchanged. The outer `main().catch(...)` handles
any _other_ startup failure (e.g. `connect()` rejecting, or `buildRegistry()` throwing — none of
the 9 real adapter factories throw synchronously in practice, but a future one could) with its own
single stderr line, then `process.exit(1)`. Stdout is never touched by diagnostics anywhere in this
file or its dependencies (R-9/§7.3) — verified by a manual
stdio smoke test (initialize → tools/list → tools/call `onchain_ping`) against both `tsx
src/index.ts` and the built `dist/index.js`: stderr stayed empty, stdout carried exactly the three
JSON-RPC response lines.

### test/env.test.ts

Unit tests for `src/env.ts` (task 001-3, closes R-6/R-12; hardened task 001-4, adversarial
multi-critic review — 9 MINOR findings, this file addresses two of them; extended task 003-6,
R-23). Explicit `vitest` imports throughout (`import { describe, it, expect, vi, afterEach } from
'vitest'`) — no `globals: true` in any vitest config, since the package `tsconfig.json`'s
`"types": ["node"]` deliberately does not add vitest's ambient globals (would need a second `types`
entry, out of scope for this package's toolchain — see the `tsconfig.json` note below). Covers:
`EnvSchema.parse({})` doesn't throw (R-12 empty-env contract); a valid `LOG_LEVEL` is accepted;
unknown keys (`PATH`, `HOME`, ...) are stripped, not rejected (`z.object()` default, not
`.strict()`); `LOG_LEVEL: ''` parses to `undefined` (001-4 fix — a blank optional env var behaves
as absent, not an invalid enum value); `loadEnv({})` mirrors the empty-env contract (an explicit
`{}` argument skips the `process.loadEnvFile()` branch, so the test never touches the real process
env or filesystem); an invalid `LOG_LEVEL` makes `loadEnv` throw, and the thrown message **and**
the `console.error` stderr line each contain the key name (`LOG_LEVEL`) but never the invalid value
string (asserted via a `vi.spyOn(console, 'error')` capture) — the D10 "never log values"
invariant, made concrete and regression-tested.

**Task 003-6 additions (R-23), 6 new cases in the `EnvSchema` describe block:** `parse({})` still
doesn't throw with the 4 new keys declared; `COINGECKO_API_KEY`/`DUNE_API_KEY`/`DATA_DIR` accept
plain strings; `ONCHAIN_PG_URL` accepts a realistic Supabase-shaped DSN (percent-encoded password
character + query string) — the exact string this task's dev-time zod check confirmed `.url()`
accepts; `ONCHAIN_PG_URL` rejects a non-URL garbage value with an error naming the key, never the
value (mirrors the existing `LOG_LEVEL` D10 pattern, direct `EnvSchema.parse()` call rather than
through `loadEnv`); `ONCHAIN_PG_URL: ''` and `COINGECKO_API_KEY`/`DUNE_API_KEY`/`DATA_DIR: ''` all
parse to `undefined` (the generalized `emptyAsUndefined` idiom, proven per-key) — these two cases
are the ones that were genuinely RED against this task's Phase 1 stub (see `src/env.ts`'s own
docstring above for the captured failure).

**`loadEnv()` NO-ARG path (001-4):** two tests spy on `process.loadEnvFile` itself
(`vi.spyOn(process, 'loadEnvFile')`) — rather than touching a real `.env` file on disk — to
exercise the branch skipped entirely by every `loadEnv({...})` call above. `vi.stubEnv('LOG_LEVEL',
'info')` (restored via `vi.unstubAllEnvs()` in `afterEach`) pins the real `process.env` to a
known-valid value first, so `EnvSchema.safeParse(process.env)` always succeeds regardless of a
developer's actual shell — these two tests are only about the `.env`-load-error handling path, not
parse outcomes. An `{ code: 'ENOENT' }`-shaped thrown error is swallowed silently: `loadEnv()`
doesn't throw and `console.error` is never called (the "missing `.env` is expected in M0" contract).
An `{ code: 'EACCES' }`-shaped thrown error does **not** abort startup either, but does warn to
stderr exactly once — asserted with an **exact** string match (not `.toContain`) against the
static prefix plus the OS error's own `.message`, which is the strongest test available that no
env VALUE could have leaked into the warning (D10).

**Real `.env` load-and-apply happy path (task 001-5, adversarial cycle 2, F-4):** a nested
`describe('loading a real .env file from disk', ...)` with its own local `beforeEach`/`afterEach`
(isolated from the parent `loadEnv` describe's hooks, so the cwd change below never leaks into the
sibling tests above) creates a temp dir (`mkdtempSync`) containing a real `.env` file
(`LOG_LEVEL=warn\n`), snapshots and deletes `process.env.LOG_LEVEL` (since
`process.loadEnvFile()` — exercised here with **no stub**, unlike the two tests above — mutates
`process.env` directly), then `process.chdir()`s into the temp dir (`process.loadEnvFile()` with
no path argument reads `.env` from `process.cwd()`). Asserts `loadEnv().LOG_LEVEL === 'warn'`.
`afterEach` restores the original cwd and `LOG_LEVEL`, then removes the temp dir
(`rmSync(..., { recursive: true, force: true })`). (Empirically verified `process.chdir()` works
under this project's vitest runner — no `ERR_WORKER_UNSUPPORTED_OPERATION`, which Node throws for
`process.chdir()` inside a `worker_threads` Worker.)

### test/e2e.stdio.test.ts

**E2E over real stdio** (task 001-3, closes R-6, confirms R-9/R-10 — the mandatory automated
stdio E2E, ARCHITECTURE §3.2/§10.2; hardened task 001-4, adversarial multi-critic review). Spawns
`src/index.ts` as a child process via `tsx`'s CLI entry, invoked as `process.execPath` running an
absolute path to `tsx/dist/cli.mjs` (not the `node_modules/.bin/tsx` shim), so the spawn never
depends on shebang execution or `PATH` resolution order, identical locally and in CI; never spawns
`dist/index.js` (ARCHITECTURE §10.2: CI runs `test` **before** `build`, so `dist/` may not exist
yet — verified during this task by `rm -rf dist` immediately before `pnpm test`). Connects with
the SDK's `Client` + `StdioClientTransport`. **Extended task 003-7 (R-20/F-1):** `tools/list` now
asserts exactly **5** tools (`onchain_ping` + the 4 new M1 tools, matched by name, sorted) — this
suite still calls ONLY `onchain_ping` over the wire; the 4 new tools' fixture-backed `registry`
injection is in-process-only (unreachable across this spawned child process boundary,
ARCHITECTURE.md §3.2 F-1) — calling them here would mean a REAL, network-capable registry
answering under spawn, exactly the live-network dependency R-21 forbids.
`test/e2e.inprocess.test.ts` (below) is what actually exercises the 4 new tools.

**`DATA_DIR` override (task 003-7, filesystem hygiene):** since `index.ts`'s `main()` now
unconditionally builds the REAL `CapabilityRegistry` (all 9 real adapters + the real two-level
cache) before registering any tool — even though this suite only calls `onchain_ping` —
`connect()` now spawns with `env: { ...process.env, DATA_DIR: <mkdtempSync temp dir> }` (removed in
`afterEach`), so the spawned server's `SqliteCacheStore` (which touches disk immediately on
construction, task 003-3) never creates/opens the REAL `~/.onchain-intel/cache.sqlite3`. Verified:
`ls ~/.onchain-intel` stays absent through a full `pnpm test` run. `tools/call onchain_ping` still
returns `structuredContent` that validates against
`PingOutputSchema`, with `version` compared against the real `package.json` `version` field (read
in the test via `readFileSync` — never a hardcoded `'0.1.0'` literal, so this test doesn't
silently go stale on a version bump); **the `content[0]` text block deep-equals `structuredContent`**
(001-4 — `content[0].type === 'text'` and `JSON.parse(content[0].text)` `toStrictEqual`
`structuredContent`, proving the two response representations can't drift apart; narrowing
`callTool`'s return type required an explicit cast to the SDK's own `CallToolResult` type — a
`'content' in result` guard can't narrow the union away, because both members carry a
`[x: string]: unknown` index signature, which defeats `in`-narrowing); a call with an unexpected
argument key (`PingInputSchema` is `.strict()`) returns `isError: true` rather than throwing or
hanging (confirmed by reading the installed SDK's `server/mcp.js`: input-validation `McpError`s
are caught server-side and converted to a `CallToolResult` with `isError: true`, not propagated as
a protocol-level error). Every test has a generous-but-bounded timeout (15s connect + 10s call) so
a stdout-framing regression (§7.3 — anything other than the MCP protocol touching stdout) fails
loudly instead of hanging CI. Teardown: `afterEach` always calls `client.close()`, which (per the
installed SDK's `client/stdio.js`) ends the child's stdin and escalates to `SIGTERM` then
`SIGKILL` if the process doesn't exit on its own — verified empirically (`ps aux | grep tsx`
immediately after a full `pnpm test` run, including a run where several assertions were
deliberately failing) showed no leaked child processes.

**Teardown self-sufficiency (001-4):** `connect()`'s outer-scope `client` variable is now assigned

### tsup.config.ts

Build config for the package: `entry: src/index.ts`, `format: esm`, `target: node22`, `clean:
true`. **`dts: false`** (declarations are NOT emitted by tsup here) — deliberate, not an
oversight: tsup 8.5.1 bundles its own dts pipeline (rollup-plugin-dts's logic, inlined in
`tsup/dist/rollup.js`), which unconditionally injects `baseUrl: compilerOptions.baseUrl || "."`
into the internal TS program it spins up for bundling `.d.ts` output. This breaks under **both**
TypeScript majors we tried: npm-latest `typescript@7.0.2` throws `TypeError: Cannot read
properties of undefined (reading 'useCaseSensitiveFileNames')` inside rollup-plugin-dts's bundled
TS-7 code path; the ecosystem-supported `typescript@6.0.3` (pinned, see package.json note above)
instead throws `error TS5101: Option 'baseUrl' is deprecated` (TS6 promotes the injected `baseUrl`
to a hard error in prep for its removal in TS7) — neither is fixable from our tsconfig since we
don't set `baseUrl` ourselves; tsup injects it internally. **Fix:** declarations are emitted as a
second `build` step via plain `tsc --emitDeclarationOnly` (reads tsconfig.json's `outDir: dist`,
so `.d.ts` lands next to tsup's `.js` output) — no rollup-plugin-dts involved, no crash. Verified
empirically: throwaway `src/index.ts` (`export const x: number = 1;`) → `pnpm build` → exit 0,
`dist/index.js` (tsup) + `dist/index.d.ts` (tsc, contents `export declare const x: number;`) both
present; throwaway `src/`/`dist/` removed again before handoff to 001-2. References `src/index.ts`,
which does not exist until 001-2 adds it — `pnpm build` is not part of 001-1's acceptance gate
(only `pnpm install` / `pnpm lint` / `pnpm format:check` are), but the pipeline is pre-verified
working so 001-2 doesn't inherit a broken build step.
