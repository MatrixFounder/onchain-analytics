# Directory: packages/mcp-server/

## Purpose

The one workspace package created in M0: the `@onchain-intel/mcp-server` package ‚Äî a stdio-only
MCP server skeleton (D3/D5, ADR-001) with an env module (zod, all keys optional in M0, R-12) and
one dummy tool, `onchain_ping` (R-10), built strictly per ARCHITECTURE.md ¬ß3.2/¬ß4.1/¬ß5.2 (task
001-2). `test/` (unit + stdio E2E, task 001-3) makes the M0 exit-criterion ‚Äî "`onchain_ping`
answers over stdio" ‚Äî verifiable in CI (R-6) instead of only by manual smoke test.

**Task 003-6 (R-23/R-24)** extended `src/env.ts`'s `EnvSchema` with 4 M1-optional keys
(`COINGECKO_API_KEY`, `DUNE_API_KEY`, `ONCHAIN_PG_URL`, `DATA_DIR` ‚Äî still no required key,
`EnvSchema.parse({})` keeps succeeding) and added the first-ever import of `@onchain-intel/core`
into this package (`test/env-degradation.integration.test.ts`, exercising `loadEnv()` together with
`@onchain-intel/core`'s `CapabilityRegistry`/`createPgHistoryAdapter` to prove the missing-key
degradation contract end-to-end). No MCP tool/handler wiring yet at that point ‚Äî `createServer`,
`server.ts`, and `src/tools/*.ts` were unchanged.

**Task 003-7 (R-16..R-20, closes TASK-003's M1 tool scope)** finishes the wiring 003-6 deferred:
4 new provider-backed MCP tools (`src/tools/get-token.ts`/`wallet-balances.ts`/`new-pairs.ts`/
`protocol-tvl.ts`, plus a shared `resolve-capability.ts` helper), `server.ts`'s `createServer` now
takes an injectable `registry?: CapabilityRegistry`, and `index.ts` builds the ONE real,
network-capable `CapabilityRegistry` (all 9 real `@onchain-intel/core` adapters + the real
two-level cache) exactly once at startup. `onchain_ping`/`ping.ts` are unchanged (R-20). See the
per-file sections below for the full design.

## Files

### src/env.ts

`EnvSchema` (zod) ‚Äî every key **optional** (R-12 in M0, R-23 for the 4 new M1 keys added this
task; `EnvSchema.parse({})` has never thrown and still doesn't). Deliberately not `.strict()`: the
real input is `process.env`, which carries hundreds of unrelated keys ‚Äî zod's default object
behavior strips them instead of rejecting them.

`LOG_LEVEL?: 'debug'|'info'|'warn'|'error'` (reserved, unused) ‚Äî M0.

**Task 003-6 (R-23) added 4 M1 keys, all optional, ARCHITECTURE.md ¬ß3.2/¬ß10.3:**
`COINGECKO_API_KEY?: string`/`DUNE_API_KEY?: string` (read by `@onchain-intel/core`'s
`coingecko`/`dune` adapters ‚Äî this schema only validates presence/shape, never reads or forwards
the value itself), `ONCHAIN_PG_URL?: string` (the `pg-history` read-only Postgres DSN, validated
with `z.string().url()`), `DATA_DIR?: string` (the cache directory override
`@onchain-intel/core`'s `resolveDataDir()` already reads, task 003-3). None of the 4 is required ‚Äî
an empty `.env` remains a fully valid, working configuration (UC-1/R-27); each capability that
needs one of them degrades explicitly through `@onchain-intel/core`'s own `isAvailable()` contract
(003-2/003-5) rather than this schema rejecting the empty env.

**`ONCHAIN_PG_URL` zod `.url()` ‚Äî empirically verified, not guessed (ARCHITECTURE.md ¬ß11 dev-time
check):** zod 4.4.3's WHATWG URL parsing accepts a realistic `postgres://user:pass@host:5432/db`
connection string, including a percent-encoded special character in the password
(`postgres://user:p%40ss@host:5432/db`) and a trailing query string
(`?sslmode=require`) ‚Äî verified via a throwaway `tsx -e` one-liner (this task's own acceptance
command) before committing to `.url()` over a hand-rolled prefix/regex check. No fallback was
needed.

**Empty-string idiom, generalized (task 003-6):** the private `emptyAsUndefined<T>(schema)` helper
wraps `z.preprocess((v) => (v === '' ? undefined : v), schema)` ‚Äî the exact fix task 001-4 applied
to `LOG_LEVEL` alone, now shared by all 5 keys (implementation choice, not an architectural one:
avoids repeating the same preprocess lambda 5 times; `LOG_LEVEL`'s own behavior is unchanged, only
its declaration is now expressed via the shared helper). A shell exporting `KEY=` (empty string, not
absent) behaves identically to the var being unset entirely, for every key in this schema.

**Stub-first evidence (task 003-6):** Phase 1 declared the 4 keys as bare `z.string().optional()`
(`ONCHAIN_PG_URL` as `z.string().url().optional()`), with NO `emptyAsUndefined` wrapper yet ‚Äî this
made the file typecheck clean (`tsc --noEmit`, the Phase 1 gate) while two new
`test/env.test.ts` cases (asserting `KEY: ''` parses to `undefined`) genuinely failed red (captured
before the fix: `ONCHAIN_PG_URL: ''` threw a `ZodError` since `''` isn't a valid URL; the 3 plain
string keys silently kept `''` instead of `undefined`). Phase 2 added the `emptyAsUndefined` wrapper
to all 5 keys, turning both red cases green with no other changes.

`loadEnv(raw?: NodeJS.ProcessEnv): Env` ‚Äî when `raw` is omitted (the real `index.ts` call site),
first tries `process.loadEnvFile()` (Node's built-in `.env` loader, stable ‚â•20.12 ‚Äî no `dotenv`
dependency; a missing `.env` is expected in M0 and silently ignored, `ENOENT` is swallowed via a
local `isEnoent()` guard; any other load error is logged to stderr but does not abort). Then
`EnvSchema.safeParse(...)`: on failure, writes `onchain-intel-mcp-server: invalid environment
configuration for: <KEY, KEY, ...>` to **stderr** (key names only, collected from
`error.issues[].path` ‚Äî **never the value**, D10) and throws; `loadEnv` itself never calls
`process.exit`, so it stays a plain, unit-testable function ‚Äî `index.ts` decides how to turn the
throw into a process exit.

### src/tools/ping.ts

`PingInputSchema = z.object({}).strict()` / `PingOutputSchema = z.object({ ok: z.literal(true),
service: z.literal('onchain-intel-mcp-server'), version: z.string(), ts: z.number().int() })` ‚Äî
the single source of truth for both runtime validation and the MCP tool-schema (no hand-written
JSON-Schema duplicate).

`pingHandler(input, ctx: { version })` ‚Äî pure function, unit-testable without a transport; returns
`PingOutputSchema.parse({ ok: true, service: 'onchain-intel-mcp-server', version: ctx.version, ts:
Date.now() })`.

`registerPingTool(server, ctx: { version })` ‚Äî calls `server.registerTool('onchain_ping', {
description, inputSchema: PingInputSchema, outputSchema: PingOutputSchema }, cb)`. `registerTool`
(not the deprecated `tool()`) was picked by reading the installed SDK's `server/mcp.d.ts`
directly (vendor-drift guard, ARCHITECTURE ¬ß11) ‚Äî it accepts a **full zod schema** as
`inputSchema`/`outputSchema` (its `AnySchema` union includes `z4.$ZodType`, which `z.object(...)`
satisfies), not just a raw shape, so `PingInputSchema`/`PingOutputSchema` are passed straight
through, unmodified. The callback returns both `content` (a JSON-stringified text block) and
`structuredContent` (the parsed object) ‚Äî the SDK requires `structuredContent` whenever
`outputSchema` is set (verified by reading `mcp.js`'s `validateToolOutput`).

### src/tools/resolve-capability.ts (new, task 003-7)

Shared `registry.resolve()` wrapper for the 4 new M1 tools ‚Äî extracted because all 4 handlers need
the byte-identical try/catch + `_meta.cache`-shape logic (DRY, developer-guidelines ¬ß1.6 "internal
abstraction ‚Äî apply professional judgment"; mirrors `@onchain-intel/core`'s own precedent of
sharing small helpers across sibling adapter files, e.g. `not-implemented-error.ts`/
`dash-metrics.ts`). `onchain_ping` does NOT use this (R-20, unchanged).

`CacheMeta { status: 'hit'|'miss'; ageMs?: number; provider: string; capability: string }` ‚Äî the
`_meta.cache` shape every one of the 4 new tools attaches to its MCP response (ARCHITECTURE.md
¬ß3.2/¬ß5.1, R-15), deliberately a sibling of `structuredContent`/`content`, never folded into
either. `ResolveOutcome = {ok: true; output: unknown; cache: CacheMeta} | {ok: false; reason:
string}`. `resolveCapability(registry, capability, chain, args)` calls `registry.resolve(...)`,
never throws: any rejection (in practice always `CapabilityUnavailableError`, ARCHITECTURE.md
¬ß9.1) becomes `{ok: false, reason: error.message}` ‚Äî that error's own constructor already builds a
message with no secret values (D10), forwarded verbatim.

### src/tools/get-token.ts, wallet-balances.ts, new-pairs.ts, protocol-tvl.ts (new, task 003-7, R-16..R-19)

Same pattern as `ping.ts`: zod in/out schema (single source of truth for both runtime validation
and the MCP tool-schema) + a pure handler (async here, since it calls `registry.resolve()`; still
unit-testable without a transport, `test/tools/*.test.ts`) + a `registerXTool(server, {registry})`
wiring function. **All 4 input schemas narrow `chain` to `z.enum(['ethereum', 'solana'])`** ‚Äî NOT
the full `ChainSchema` (task 003-7 reviewer note, Major-2 from architecture review cycle 1):
`isValidAddress`/`normalizeAddress` don't implement Dash address validation (`dash-platform` works
through `Snapshot`, not `Token`/`Wallet`), so accepting `'dash'` here would be a value that always,
unconditionally fails validation ‚Äî a misleading contract. `new-pairs.ts`/`protocol-tvl.ts` each
declare this enum ONCE as a local `SUPPORTED_CHAIN` constant, reused for both their input's and
output's `chain` field (keeps the task's own acceptance grep ‚Äî "narrowed in all 4 tool files" ‚Äî at
exactly 4 matches across `src/tools/`, one per file, instead of also matching output-schema/
docstring repeats).

- **`get-token.ts`** (R-16) ‚Äî `GetTokenInputSchema` (`chain` + `address: z.string().min(1)`,
  `.strict()`, `superRefine` ‚Üí `isValidAddress(chain, address)`). Output is `TokenSchema`
  re-exported from `@onchain-intel/core` verbatim (task scope: "outputs = canonical types from
  `@onchain-intel/core`") ‚Äî no new zod schema for a shape that already has one. Capability:
  `token.price` (switched from `token.metadata` in adversarial cycle 3 ‚Äî both route to `coingecko`
  and its `normalize()` produces the byte-identical `Token` either way, but the combined payload is
  cached under its most VOLATILE constituent's TTL: 60s price, not 3600s metadata, else an
  hour-stale `priceUsd` is legal; `token.metadata` route stays for future metadata-only consumers). `getTokenHandler` re-normalizes
  `input.address` via `normalizeAddress` before it becomes part of the cache-key `args` (the
  documented `net/args-hash.ts` contract: args must be the normalized, post-zod-validation tool
  input). **Bounded address length (adversarial cycle 2, finding 3):** `address` gained
  `.max(MAX_ADDRESS_LENGTH)` (64 ‚Äî a real EVM address is <=42 chars, a real Solana base58 pubkey is
  <=44). **Empirically verified this schema's own `superRefine` still runs even after `.max()`
  already flagged an issue** (zod keeps collecting every issue it can here, it doesn't abort
  early) ‚Äî so `superRefine` ALSO checks `val.address.length > MAX_ADDRESS_LENGTH` at its own top
  and returns immediately in that case, which is what actually guarantees the expensive
  `isValidAddress`/`bs58.decode` work (quadratic-ish for very long inputs) is skipped entirely for
  a pathological address, not merely "eventually rejected after also doing the expensive work".
  One new test: a 100k-character address rejects in well under 100ms.
- **`wallet-balances.ts`** (R-17) ‚Äî `WalletBalancesInputSchema` is the literal sample from
  ARCHITECTURE.md ¬ß5.1 (same `chain`+`address`+`superRefine` shape as `get-token.ts`, including the
  same `.max(MAX_ADDRESS_LENGTH)` + length-guarded `superRefine`, adversarial cycle 2, finding 3).
  Output is `WalletSchema` verbatim; capability `wallet.balances.native`, backed by
  `rpc-evm`/`rpc-solana` per chain (OQ-1).
- **`new-pairs.ts`** (R-18) ‚Äî `NewPairsInputSchema` (`chain` + `limit?:
z.number().int().positive().optional()`). Output is a NEW zod schema, `NewPairsOutputSchema =
{chain, pairs: Pool[], source, fetchedAt}` (copied literally from ARCHITECTURE.md ¬ß5.1) ‚Äî not one
  of the six canonical `types/*` schemas (only `Pool`, the array element type, is); mirrors
  `defillama`'s own `ProtocolTvlResult` precedent (a tool-contract-level shape, not a canonical
  domain entity). Capability `pairs.new` ‚Üí `dexscreener`, which returns a bare `Pool[]`; the
  handler wraps it: `source` = the answering adapter id (`outcome.cache.provider`), `fetchedAt` =
  `Date.now()` at response-build time (a documented implementation choice, developer-guidelines
  ¬ß1.5 ‚Äî distinct from each individual `Pool`'s OWN `fetchedAt`, which may be older on a cache hit).
  **Cache-key split fix (post-M1 polish, cheap-fix backlog item 1):** an omitted `limit` used to
  build `args = {chain}` while an explicit `limit: 10` (`dexscreener`'s own `DEFAULT_LIMIT` ‚Äî
  `pairs.new`'s only registered adapter) built `args = {chain, limit: 10}` ‚Äî two DIFFERENT
  `deriveArgsHash` cache keys for the identical logical query, so the two shapes never shared a
  cache entry (a duplicate upstream `dexscreener` fetch for what a caller would reasonably expect
  to be one cached query). `newPairsHandler` now materializes `input.limit ?? DEFAULT_LIMIT` (a
  local constant, duplicated from ‚Äî not re-exported from ‚Äî `dexscreener`'s own internal default;
  kept in sync manually rather than widening `@onchain-intel/core`'s public export surface for one

  new `test/tools/new-pairs.test.ts` case: a real two-level cache (`createCacheStore({dbPath:

  explicit-`limit: 10` call is a cache HIT on the second call, with the adapter's `fetch()` only
  ever invoked once.
  **De-duplicated validation (adversarial cycle 1, fix I):** `newPairsHandler` used to run a
  redundant standalone `z.array(PoolSchema).parse(outcome.output)` BEFORE building `output` via
  `NewPairsOutputSchema.parse(...)` ‚Äî the latter's own `pairs` field is already `z.array(PoolSchema)`,
  so `outcome.output` (the adapter's raw `Pool[]`) was validated against the same schema twice.
  `outcome.output` is now passed straight into `NewPairsOutputSchema.parse({..., pairs:
outcome.output, ...})` ‚Äî validated exactly once. The other 3 new tools (`get-token.ts`/
  `wallet-balances.ts`/`protocol-tvl.ts`) were checked for the same pattern and don't have it ‚Äî

- **`protocol-tvl.ts`** (R-19) ‚Äî `ProtocolTvlInputSchema` (`chain` + `protocolSlug:

  no real DeFiLlama protocol slug is anywhere near that long; this rejects a pathologically long
  input (e.g. a 10k-character string) at the schema layer, cheaply, before it could otherwise reach

  FIRST zod schema for `defillama`'s `ProtocolTvlResult` shape (`{protocol, chain, tvlUsd,

  documented choice) ‚Äî `tvlUsd`/`totalTvlUsd` are `.nonnegative()`, mirroring

  `protocol.tvl` ‚Üí `defillama`. **`safeParse`, never `parse` (adversarial cycle 2, finding 1a):**

  potentially-multi-issue zod-error dump. The matching enforcement on the PRODUCER side ‚Äî
  `defillama`'s own `normalize()` now rejects a non-finite/negative `tvlUsd`/`totalTvlUsd` before

  finding 1b). One new test: a fake adapter returning `tvlUsd: -1` makes `protocolTvlHandler`

**Shared `isError` wiring (all 4, task 003-7 reviewer note):** each `registerXTool`'s callback
checks `outcome.ok` explicitly ‚Äî `false` ‚Üí `{isError: true, content: [{type: 'text', text:

### src/server.ts (extended, task 003-7)

`createServer(deps: { env: Env; version: string; registry?: CapabilityRegistry }): McpServer` ‚Äî
`registry` is now injectable (ARCHITECTURE.md ¬ß3.2/¬ß5.2, F-1): registers all 5 tools (`ping` +
the 4 new M1 tools), each of the 4 new ones given `{registry}` as context. `deps.registry`'s
fallback (`new CapabilityRegistry(routes, new Map())`, real `routes` table from
`@onchain-intel/core`'s `providers.config.ts`, an EMPTY adapter `Map`) is a deliberately INERT
default ‚Äî every capability degrades gracefully (`CapabilityUnavailableError`, "no adapter
registered for this id") rather than crashing, but it is NEVER the real, network-capable registry:
assembling all 9 real adapters + the real cache is `index.ts`'s single, explicit responsibility
(this task's own instruction, ARCHITECTURE.md ¬ß3.2 "—Å—Ç—Ä–æ–∏—Ç—Å—è –æ–¥–∏–Ω —Ä–∞–∑ –≤ index.ts") ‚Äî `index.ts`
always constructs one and passes it in explicitly. This fallback exists purely so `createServer`
stays type-safe and harmless if a future caller ever omits `registry` outside that documented
production path (implementation choice, developer-guidelines ¬ß1.5). This injection point is the
ONLY mechanism "MCP E2E without network" (R-21) relies on ‚Äî no global `fetch` mock, a different
implementation of the same public `CapabilityRegistry.resolve()` contract is injected here; it
works ONLY in-process (unreachable across a spawned child process boundary, F-1) ‚Äî see
`test/e2e.inprocess.test.ts` below.

### src/index.ts (extended, task 003-7)

Bin entry (`#!/usr/bin/env node`, preserved by tsup/esbuild in `dist/index.js`, which is also
marked executable). Reads `version` **once** via `createRequire(import.meta.url)` +
`require('../package.json')` (cast to a local `PackageJson` interface ‚Äî never left as `any`) ‚Äî
chosen over a JSON import attribute (`with { type: 'json' }`) because that syntax is flaky under
the pinned TypeScript 6 / NodeNext combination this package uses; `createRequire` + `require(...)`
works identically under `tsx src/index.ts` (dev) and the bundled `dist/index.js` (both sit one
directory below the package root, so `../package.json` resolves the same way in both). Note for
ESLint: `@typescript-eslint/no-require-imports` (part of the shared `recommended` config) does
**not** flag this ‚Äî the rule explicitly whitelists a `require()` call when `require` resolves to a
locally-declared variable (as `createRequire` produces), rather than the CJS global.

**`buildRegistry(env: Env): CapabilityRegistry` (new, task 003-7)** ‚Äî assembles the ONE real,
network-capable registry for production: a `Map` with all 9 real `ProviderAdapter`s from
`@onchain-intel/core` (`createCoingeckoAdapter({env})`/`createPgHistoryAdapter({env})` ‚Äî the two
that read an optional env key ‚Äî plus the other 7, all keyless/no-arg), the real `routes` table,
and the real cache (`createCacheStore()`, no explicit `dbPath` ‚Äî its `resolveDataDir()` default
already reads `process.env.DATA_DIR`, which `loadEnv()` (called just before this) has already
synced into `process.env` via `process.loadEnvFile()`). This is the single place the real
9-adapter set is ever constructed (task's own instruction) ‚Äî `server.ts`'s own `registry` default
is a separate, deliberately inert fallback (see its section above).

`main()`: `loadEnv()` (its own try/catch ‚Äî on throw, `process.exit(1)` immediately, no extra
logging, since `loadEnv` already wrote the stderr diagnostic) ‚Üí `buildRegistry(env)` ‚Üí
`createServer({ env, version, registry })` ‚Üí `await server.connect(new StdioServerTransport())` ‚Äî
the **only** place a transport is chosen (R-9); a future (M6) alternative HTTP-based transport
would be attached here too, `createServer` stays unchanged. The outer `main().catch(...)` handles
any _other_ startup failure (e.g. `connect()` rejecting, or `buildRegistry()` throwing ‚Äî none of
the 9 real adapter factories throw synchronously in practice, but a future one could) with its own
single stderr line, then `process.exit(1)`. Stdout is never touched by diagnostics anywhere in this
file or its dependencies (R-9/¬ß7.3) ‚Äî verified by a manual
stdio smoke test (initialize ‚Üí tools/list ‚Üí tools/call `onchain_ping`) against both `tsx
src/index.ts` and the built `dist/index.js`: stderr stayed empty, stdout carried exactly the three
JSON-RPC response lines.

TS-7 code path; the ecosystem-supported `typescript@6.0.3` (pinned, see package.json note above)
instead throws `error TS5101: Option 'baseUrl' is deprecated` (TS6 promotes the injected `baseUrl`
to a hard error in prep for its removal in TS7) ‚Äî neither is fixable from our tsconfig since we
don't set `baseUrl` ourselves; tsup injects it internally. **Fix:** declarations are emitted as a
second `build` step via plain `tsc --emitDeclarationOnly` (reads tsconfig.json's `outDir: dist`,
so `.d.ts` lands next to tsup's `.js` output) ‚Äî no rollup-plugin-dts involved, no crash. Verified
empirically: throwaway `src/index.ts` (`export const x: number = 1;`) ‚Üí `pnpm build` ‚Üí exit 0,
`dist/index.js` (tsup) + `dist/index.d.ts` (tsc, contents `export declare const x: number;`) both
present; throwaway `src/`/`dist/` removed again before handoff to 001-2. References `src/index.ts`,
which does not exist until 001-2 adds it ‚Äî `pnpm build` is not part of 001-1's acceptance gate
(only `pnpm install` / `pnpm lint` / `pnpm format:check` are), but the pipeline is pre-verified
working so 001-2 doesn't inherit a broken build step.

### .prettierignore

Package-local, added in 001-2 (`dist/` only). **Genuinely forced, empirically verified:** once
`pnpm build` produces `dist/` (esbuild + `tsc --emitDeclarationOnly` output, never hand-formatted),
`prettier --check .` run from this package's directory started flagging `dist/*.js`/`*.d.ts` as
unformatted and failing. Unlike `.prettierrc` (resolved via cosmiconfig's upward directory search
‚Äî see the Notes below), Prettier's `.prettierignore` lookup is **CWD-relative only**: a root-level
`../../.prettierignore` (also added in 001-2, mirroring `eslint.config.js`'s `ignores`) is not
found by a script invoked as `pnpm --filter @onchain-intel/mcp-server run format:check` (cwd =
this package). Verified: `prettier --check . --ignore-path ../../.prettierignore` succeeds (proves
the root file's content is correct), but plain `prettier --check .` does not pick it up ‚Äî hence
this package-local file is required in addition to the root one. Future `packages/*` (M1+) will
each need their own copy of this same one-line file until/unless the `format:check` script itself
is changed to pass `--ignore-path` explicitly (out of scope for 001-2 ‚Äî no script changes made).
