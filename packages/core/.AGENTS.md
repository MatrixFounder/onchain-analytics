# Directory: packages/core/

## Purpose

`@onchain-intel/core` — the M1 library package (ARCHITECTURE.md §3.1/§3.2, D5/D12 "packages/core
split"): canonical zod domain types (`Token`, `Wallet`/`Balance`, `Pool`, `OHLCV`, `Snapshot`,
`ChainSchema`), the chain/address normalization module (`normalizeAddress`/`isValidAddress`) —
both task 003-1, R-1/R-2 — the hot-swappable provider-adapter contour: `ProviderAdapter`/
`CapabilityRegistry`/`CacheStore` seam (`src/adapters/*`), the declarative `providers.config.ts` (9
adapter registrations + capability routes, copied literally from ARCHITECTURE.md §3.2), the SSRF
gate + rate-limiter + canonical args-hash (`src/net/*`) — all task 003-2, R-3/R-4/R-25/R-26 — and,
added this task (003-3, R-13/R-14/R-15), the real two-level cache (`src/cache/*`): `lru-cache` hot
layer + `better-sqlite3` persistent layer in `DATA_DIR`, implementing the exact `CacheStore`
interface 003-2 defined, plus hit/miss stats. Consumed by `@onchain-intel/mcp-server` via a
`workspace:*` dependency (declared task 003-1; not yet imported — that starts at task
003-6/003-7). **Task 003-4 (R-5/R-6/R-7) added the first three LIVE provider adapters** —
`coingecko`, `dexscreener`, `defillama` (`src/adapters/<id>/index.ts`), each a factory function
implementing `ProviderAdapter` for real, plus their committed fixtures + golden `normalize()`
tests. **Task 003-5 (R-8..R-12) completes the full nine-adapter set**: `rpc-evm`/`rpc-solana`
(LIVE JSON-RPC native-balance adapters, OQ-1 resolved), `platform-explorer` (LIVE REST, the only
live Dash source in M1 + its own history method), `dash-platform` (interface + fixture-contract
only, F-3 — `isAvailable()` unconditionally `false`), `dune` (interface/config-stub, R-8 amended
scope — no live call/fixture/test beyond registration+availability), and `pg-history` (optional
read-only Postgres history adapter over the new `src/pg/read-client.ts`, `pg` dependency). All
nine `providers.config.ts` ids now have a matching real `ProviderAdapter` factory. The cache
itself is real as of task 003-3, but nothing in this package constructs one by default yet:
`CapabilityRegistry`'s constructor still defaults to `PassthroughCacheStore` (unchanged, see
`src/adapters/cache-store.ts` below) — wiring `createCacheStore()` and all nine adapters into a
real `CapabilityRegistry` instance is `mcp-server`'s bootstrap responsibility (tasks 003-6/003-7),
not this package's.

### Build choice: plain `tsc`, no tsup (ARCHITECTURE.md §3.1/§3.2/§6.4)

This package's `build` script is `tsc -p tsconfig.build.json` — a single `tsc` invocation emits
both `.js` (NodeNext-native ESM output) and `.d.ts` for every file under `src/` in one step (see
the `dist/` file list below). This is a deliberate architectural decision (not a shortcut): the
`mcp-server` package needs tsup/esbuild only because it has a `bin` entry that must be bundled into
a single executable file, and separately hit an upstream tsup/rollup-plugin-dts bug when it tried
to have tsup also emit declarations (`baseUrl` injection crashes under both TS6 and TS7 — see
`packages/mcp-server/.AGENTS.md`'s `tsup.config.ts` note for the full empirical writeup), which is
why `mcp-server`'s build is a two-step `tsup && tsc --emitDeclarationOnly`. `packages/core` has no
`bin`, needs no bundling, and NodeNext's own module resolution is sufficient at runtime for a
library consumed only via `workspace:*` — so it sidesteps that whole bug class entirely rather than
reproducing it in a second package. Verified: `pnpm -r build` builds `core` (plain `tsc`) before
`mcp-server` (tsup+tsc) via pnpm's default topological ordering (`mcp-server`'s `workspace:*`
dependency on `@onchain-intel/core`) — confirmed by output ordering in a live `pnpm -r build` run,
not assumed.

`dist/` after a build: `index.{js,d.ts}`, `chain/address.{js,d.ts}`, `types/{chain,token,wallet,
pool,ohlcv,snapshot,index}.{js,d.ts}` — an exact mirror of `src/`, nothing from `test/`.

### src/chain/address.ts

`normalizeAddress(chain, raw): string` / `isValidAddress(chain, raw): boolean`
(ARCHITECTURE.md §3.2/§4.1, reviewer note — the mandatory single point of address canonicalization
used by both MCP-tool input `superRefine`s and adapters before building a cache key):

- **`ethereum`:** canonical form is the **EIP-55 checksum**, never lowercase (ADR-001 D5). Algorithm:
  strip an optional `0x`/`0X` prefix, validate the remaining 40 characters are hex
  (`/^[0-9a-fA-F]{40}$/`), lowercase them, hash that lowercase hex **string** with `keccak_256`
  (`@noble/hashes/sha3.js` — accepts a `string | Uint8Array` directly, no manual `TextEncoder`
  needed), then for each hex digit of the lowercased address: digits (`0`-`9`) pass through
  unchanged, letters (`a`-`f`) are upper-cased iff the corresponding hex nibble of the hash (same
  character index) is `>= 8`. This is a pure function of the address bytes — `normalizeAddress`
  applied to any input case (all-lower, all-upper, already-checksummed, or an arbitrary mix)
  produces the identical output, proven in `test/chain-address.test.ts`. `isValidAddress` for
  ethereum only checks the hex-format contract (length + charset) — it does **not** additionally
  require the input's _own_ casing to already be a correct checksum (that would make
  `isValidAddress` reject perfectly normalizable lowercase input, contradicting the "any case in"
  contract).
- **`solana`:** canonical form is the input **as-is** — base58 is case-sensitive (unlike hex),
  so lowercasing would corrupt the address rather than canonicalize it. Validation: `bs58.decode`
  succeeds (any character outside the base58 alphabet — `0`, `O`, `I`, `l` are excluded — throws,
  caught and turned into `false`/a thrown `Error`) **and** the decoded byte length is exactly 32
  (a raw ed25519 pubkey, no version/checksum byte, unlike Bitcoin-style base58check).
- **`dash`:** not implemented in M1 (ARCHITECTURE.md §4.1 — `dash-platform` emits `Snapshot`, not
  `Wallet`/`Balance`, so there's nothing here to normalize/validate yet). `isValidAddress('dash',
*)` unconditionally returns `false`; `normalizeAddress('dash', *)` unconditionally throws. This
  mirrors the `dash-platform` adapter's own `isAvailable()` contract (§3.2: unconditionally
  unavailable in M1, not "unavailable if unreachable") — both are explicit, not silently-broken,
  M1 boundaries.

`normalizeAddress` throws (never returns a sentinel) on invalid/unsupported input — the contract
documented in ARCHITECTURE.md §3.2: "невалидный вход → бросок, ловится вызывающим
`superRefine`/адаптером". No custom `Error` subclass was introduced (Zero-Dependency /
no-speculative-complexity — a plain `Error` with a descriptive, value-free message is sufficient
for this task's scope; callers that need typed error handling can add a subclass when they
actually need to branch on it).

### src/adapters/types.ts (new, task 003-2, R-3)

The stable internal interfaces the rest of the provider-adapter contour depends on
(ARCHITECTURE.md §3.2, D4):

- `CapabilityDescriptor { id, chains? }` — a single routable data capability; `chains` narrows it
  to specific chains, omitted means chain-agnostic.
- `ProviderAdapter { id, capabilities(), costOf(), fetch(), normalize(), isAvailable()? }` — no
  concrete adapter implements this yet (003-4/003-5); `fetch()` returns `unknown` (a
  provider-specific raw shape, narrowed only inside that adapter's own `normalize()`) — the
  anti-corruption-layer contract: `CapabilityRegistry` never sees a raw provider DTO, only what
  `normalize()` produces. `isAvailable?()` is optional — an adapter with no env/key precondition
  (e.g. a keyless REST API) omits it and `CapabilityRegistry` treats that as "always available"
  (`??  { ok: true }` at the call site in `registry.ts`).
- `AdapterRegistration { id, hosts, rateLimit, requiresEnv }` — `providers.config.ts`'s per-adapter
  row; `hosts` is the SSRF allowlist source-of-truth for **that adapter only** (R-25, never a
  merged/global list); `rateLimit` feeds `throttle` (R-26); `requiresEnv` is informational only —
  the actual availability decision is always the adapter's own `isAvailable()`.
- `CapabilityRoute { capability, chains?, adapterIds }` — `adapterIds` order is priority +
  fallback chain (R-11); changing priority is a `providers.config.ts` edit, never a call-site change.

No `any`-leaks anywhere in this file (R-3 acceptance) — verified by `tsc --noEmit` plus this
package's `eslint.config.js`-inherited `no-explicit-any` rule staying clean.

### src/adapters/cache-store.ts (new, task 003-2 — the cache seam for 003-3)

`CacheStore { get(provider, capability, argsHash): Promise<CacheGetResult | undefined>;
set(provider, capability, argsHash, value): Promise<void> }` — the interface
`CapabilityRegistry.resolve()` depends on, keyed identically to the `cache_entries` UNIQUE
constraint `(provider, capability, args_hash)` (ARCHITECTURE.md §4.2). `CacheGetResult { value,
ageMs }` — `ageMs` is forwarded verbatim by `resolve()` as its own `ageMs` field; the Registry
never inspects it to decide hit vs. miss (that's entirely the concrete store's job — TTL
expiry means `get()` returns `undefined`, whether nothing was ever written or the store decided a
written entry is stale).

**Task 003-2 built only this interface + one no-op stand-in, deliberately not the real cache (guard
R-27 at the time):** `PassthroughCacheStore` — `get()` always returns `undefined` (always miss),
`set()` discards the value — is `CapabilityRegistry`'s default third constructor argument
(`new PassthroughCacheStore()`), so a caller that doesn't inject a `CacheStore` still gets a
correctly-typed, harmless one (each `CapabilityRegistry` instance gets its own, never a shared
module singleton — ARCHITECTURE.md §8 "factory, not singleton"). **`PassthroughCacheStore` is still
the default today** — task 003-3 (below, `src/cache/*`) landed the real `SqliteCacheStore`/
`TwoLevelStore`/`createCacheStore()` implementing this exact interface, but did **not** change this
default: `registry.ts` is unmodified (still zero changes since 003-2), and a future `mcp-server`
bootstrap (tasks 003-6/003-7) is what will actually construct `new CapabilityRegistry(routes,
adapters, createCacheStore())` in place of relying on the default. The seam itself is exercised
end-to-end by `test/registry.test.ts`'s `FakeCacheStore` double (proves hit/miss/`set()`-call-shape
against the exact same interface 003-3's real store implements) — see the `src/cache/*` section
further down for the concrete implementation.

### src/adapters/registry.ts (task 003-2, R-4/R-11/R-24 — completes 003-1's earlier stub)

`CapabilityUnavailableError extends Error { capability, chain, tried: CapabilityAttempt[] }` —
thrown when every adapter in a route's `adapterIds` was unavailable/failed, or no route matches
`(capability, chain)` at all (`tried: []` in that case). `CapabilityAttempt { adapterId, reason }`
records one skipped/failed attempt (R-24 explicit degradation — never a silent empty result).

`CapabilityRegistry` — constructor takes `(routes: CapabilityRoute[], adapters: Map<string,
ProviderAdapter>, cache: CacheStore = new PassthroughCacheStore())`; a factory, not a module
singleton (ARCHITECTURE.md §8, task reviewer note) — tests build their own small route tables +
mock-adapter Maps instead of importing the real `providers.config.ts` (which has no matching real
`ProviderAdapter` instances until 003-4/003-5).

`resolve(capability, chain, args)` algorithm (implemented this task, replacing 003-1's
`throw new Error('not implemented yet')` stub):

1. Find the route where `route.capability === capability` and (`route.chains` is unset or
   contains `chain`). No match → `CapabilityUnavailableError` with `tried: []`.
2. Compute `argsHash = deriveArgsHash(capability, args)` once per call (not per-adapter — the
   capability+args pair is the same regardless of which adapter ultimately serves it).
3. Walk `route.adapterIds` in order:
   - No `ProviderAdapter` registered for that id in the `adapters` Map → recorded as
     `{ adapterId, reason: 'no adapter registered for this id' }`, skip-to-next. This is the
     documented behavior for ids `providers.config.ts` names but no adapter implements yet (see
     that file's own docstring) — **not** a compile-time or runtime error in this package.
   - `adapter.isAvailable?.() ?? { ok: true }` reports `{ ok: false, reason }` → recorded, skip.
   - `cache.get(adapter.id, capability, argsHash)` hit → return
     `{ result: cached.value, source: adapter.id, cache: 'hit', ageMs: cached.ageMs }` immediately
     — `fetch`/`normalize` are never called on a hit.
   - Miss → `try { fetch → normalize → cache.set → return { result, source, cache: 'miss' } }
catch (error) { record { adapterId, reason: error.message }; continue to next adapterId }` —
     a throw from either `fetch()` or `normalize()` is caught and treated as a fallback trigger,
     never propagated as-is (R-11 hot-swap).
4. Every `adapterId` exhausted without a return → `CapabilityUnavailableError` with the full
   `tried` list.

Anti-corruption layer preserved: only `normalize()`'s return value is ever part of the resolved
`result` — the raw `fetch()` payload never escapes this method (asserted in
`test/registry.test.ts` via `result.result !== raw` object-identity check).

### src/net/safe-fetch.ts (task 003-2, R-25 — the single point of outgoing HTTP)

`SsrfBlockedError extends Error { hostname }` — carries only the rejected hostname, never the full
URL/query (not a secret, but no reason to carry more than needed to diagnose a misconfigured
allowlist). `assertAllowedHost(hostname, allowlist)` — throws if `hostname` isn't in `allowlist`;
transport-agnostic (designed to also serve a future non-HTTP transport, e.g. `dash-platform`'s
deferred gRPC channel — ARCHITECTURE.md §7.3 — though only `safeFetch` calls it in M1).

`safeFetch(url, opts, allowlist, fetchImpl = fetch)` — resolves the target hostname and
`assertAllowedHost`s it **before** any network attempt; calls `fetchImpl` with
`redirect: 'manual'`; on a 3xx response with a `Location` header, resolves that `Location` against
the current hop's URL (handles both absolute and relative redirects identically to a real
browser), re-checks **that** hostname against the same `allowlist` before following it, up to
`MAX_REDIRECTS = 3` hops — never trusts `Location` blindly (R-25/§7.3). `fetchImpl` is injectable
(defaults to the global `fetch`) specifically so `test/safe-fetch.test.ts` never touches the real
network — every test supplies a `vi.fn()` fake returning canned `Response`s per hop.

### src/net/rate-limit.ts (task 003-2, R-26 — per-provider token bucket)

`TokenBucketConfig { capacity, refillPerSec }` (the shape of `providers.config.ts`'s `rateLimit`
field). `createThrottle(deps?: { now?, wait? })` builds an isolated `Throttle` function
(`(providerId, config) => Promise<void>`) with its own `Map<providerId, BucketState>` — a factory,
not a shared singleton (mirrors the Registry/CacheStore "factory, not singleton" principle). Each
call: refills the named provider's bucket by `elapsedSeconds * refillPerSec` (capped at
`capacity`) based on time since the bucket's last check, then either consumes one token
immediately or awaits `deficit / refillPerSec` seconds before proceeding. `deps.now`/`deps.wait`
are injectable (production default: real `Date.now`/`setTimeout`) precisely so
`test/rate-limit.test.ts` never spends real wall-clock time or touches a real timer — tests supply
a fake clock whose `wait()` just advances an in-test counter and resolves immediately. The
module-level `throttle` export (`= createThrottle()`) is the production singleton adapters will
call in 003-4/003-5 — in-memory, per-process, no persistence (M1 — one process, ARCHITECTURE.md §8).

### src/net/args-hash.ts (task 003-2 — `cache_entries`' `args_hash`, ARCHITECTURE.md §7.2)

`canonicalize(value)` — recursively sorts object keys (arrays keep element order, only object
_keys_ are canonicalized; primitives pass through unchanged) so that `{chain, address}` and
`{address, chain}` — the same logical tool input, built in a different key order — produce
byte-identical `JSON.stringify` output. `deriveArgsHash(capability, args)` —
`sha256(hex(JSON.stringify({ capability, args: canonicalize(args) })))` via Node's built-in
`node:crypto` `createHash` (no new dependency). `args` **must always** be the normalized,
post-zod-validation, post-`normalizeAddress` tool input — **never** `process.env` or any secret
(D10/§7.2); folding `capability` into the hashed payload (not just a separate cache-key column)
prevents two different capabilities called with coincidentally-identical `args` from colliding.

003-2) — **zero changes to `src/adapters/registry.ts` itself**: the Registry already accepts any

argsHash)` exactly once per available adapter it walks, which is the one seam this task's hit/miss

- **`ddl.ts`** — `CACHE_DDL`, the final DDL string (declarative data, not logic — shipped unchanged
  since Phase 1): `providers(id, kind, notes)` + `cache_entries(id, provider, capability, args_hash,
value_json, created_at, expires_at)` + `UNIQUE(provider, capability, args_hash)` +

  file's own prose comments (they say "row-numbering feature", never the literal word the grep
  bans). Carries the R-14 "future `usage(provider, day, credits_used)` FK's onto this same

  `path.join(os.homedir(), '.onchain-intel')` — never `process.cwd()`-relative (the MCP server

  injectable purely for deterministic tests, mirroring `safeFetch`'s `fetchImpl` / `createThrottle`'s
  `deps` DI convention already established in this package (task 003-2).
- **`ttl.ts`** — `ttlFor(capability): number` (seconds), copied literally from ARCHITECTURE.md
  §3.2's TTL table (`platform.*` expanded into its four concrete capabilities). Three capabilities
  the table doesn't explicitly cover (`pool.info`, `privacy.shielded_pool.history`,
  `platform.metrics.history`) get a documented implementation decision inline (developer-guidelines
  §1.5 "implementation ambiguity") rather than a silent guess: `pool.info` shares dexscreener's
  `protocol.tvl`-like volatility bucket (300s); both `*.history` capabilities inherit their live
  counterpart's 3600s bucket, per the table's own stated rationale for that bucket. Any other,
  wholly unmapped capability string falls back to a conservative documented default (300s) instead
  of throwing.
- **`stats.ts`** — module-scoped `Map<capability, {hit, miss}>`; `recordCacheAccess(provider,
capability, 'hit'|'miss', ageMs?)` increments it **and** writes the mandatory stderr line
  (`cache=hit|miss provider=<id> capability=<cap> ageMs=<n>` — no args/secret values, NEVER
  stdout — M0 §7.3 invariant); `getCacheStats()` returns a plain-object snapshot (`_meta.cache`
  consumer, task 003-7); `resetCacheStats()` is a test-only utility (module state would otherwise
  leak across test files).
- **`lru.ts`** — `LruHotLayer`, wrapping `lru-cache`. `set(provider, capability, argsHash, value,
ttlMs)` takes TTL explicitly in **milliseconds from the caller** — never derived internally from
  `ttlFor` — specifically so `TwoLevelStore` can promote a cold hit with its REMAINING ttl (not a
  fresh full ttl), and so unit tests can exercise real expiry with a small ttl + a short real
  `setTimeout` wait instead of waiting out a capability's full real-world TTL (e.g. 60s for
  `token.price`). An entry with `ttlMs <= 0` is not stored. Composite key is
  `` `${provider}\0${capability}\0${argsHash}` `` (NUL-separated).
- **`sqlite-store.ts`** — `SqliteCacheStore implements CacheStore`. Constructor: `mkdirSync`s the
  parent dir (skipped for `':memory:'`), opens the connection, issues `PRAGMA foreign_keys = ON`
  via `db.exec('PRAGMA foreign_keys = ON;')` — a **literal SQL string via `exec()`, not
  `db.pragma('foreign_keys = ON')`** — deliberately, so the task's acceptance grep
  (`PRAGMA foreign_keys\s*=\s*ON`) finds it verbatim in the source (`db.pragma()`'s argument alone
  doesn't contain the literal word `PRAGMA`); `journal_mode = WAL` uses the ordinary `.pragma()`
  form (not grepped, and idiomatically fine either way) — re-issued on **every** open, since
  `foreign_keys` is connection-scoped, not persisted in the file (`journal_mode` IS persisted).
  Runs `CACHE_DDL`, then upserts every constructor-supplied `providers` entry (`kind`: a small
  explicit `dune`-is-the-only-`'paid'`-adapter lookup sourced from ARCHITECTURE.md §3.2's adapter
  summary table — `AdapterRegistration` itself carries no such field) BEFORE any `cache_entries`
  write can reference one as an FK target. `get()` selects by the UNIQUE key, treats `expires_at <=
now` as a miss **and deletes the stale row** (so it never shadows a later write); `set()` is
  `INSERT ... ON CONFLICT (provider, capability, args_hash) DO UPDATE SET value_json=excluded...,
created_at=excluded..., expires_at=excluded...` — an upsert, never `DO NOTHING` (DB-SCHEMA-CONCEPT
  §1.5 "aggregates" branch — a cache entry is a recomputable projection, not an append-only
  observation).
- **`two-level-store.ts`** — `TwoLevelStore implements CacheStore`: `get()` walks hot
  (`LruHotLayer`) → cold (injected persistent `CacheStore`) → miss; a cold hit is promoted into the
  hot layer with `ttlFor(capability) * 1000 - coldHit.ageMs` (the entry's REMAINING ttl); `set()`
  writes both layers (persistent first, then hot with the full capability ttl). Hit/miss stats +
  the stderr line are recorded **inside this class's `get()`**, not by an edit to
  `registry.ts` — see the section header above for why that's equivalent. `createCacheStore(options?)`
  factory: assembles a `SqliteCacheStore` (bootstrapped from `adapterRegistrations`, all 9, by
  default) wrapped in a `TwoLevelStore` — this is what a future `mcp-server` bootstrap (tasks
  003-6/003-7) constructs and injects as `CapabilityRegistry`'s third constructor argument in place
  of the default `PassthroughCacheStore`.

**Anti-corruption-layer split (task 003-4 reviewer note — "narrowing only inside `normalize()`"):**
each adapter's HTTP step returns a small **private, adapter-own hand-off shape** — never the
literal wire body alone — because `normalize(cap, raw)` (D4's fixed two-argument signature) has no
way to learn which chain the caller asked for from a raw provider DTO alone, and this adapter's
own reviewer-mandated address/chain reasoning needs it:

- `coingecko`: `{chain, raw}`. Confirmed by a live probe (2026-07-22) that the response's own
  `asset_platform_id` field is the token's **primary** platform, not necessarily the one queried
  (querying USDC's Solana contract still returns `asset_platform_id: "ethereum"`, since Ethereum
  is USDC's primary platform) — so `chain` cannot be recovered from the body alone and must be
  carried alongside it. `normalize()` reads `raw.detail_platforms[chain]` for the
  query-chain-specific `contract_address`/`decimal_place`, then re-applies `normalizeAddress`
  (the fixture's own `contract_address` is lowercase — the canonical `Token.address` must be the
  EIP-55 checksum, reviewer note) — `symbol`/`name`/`market_data.*` are chain-agnostic top-level
  fields. Both `token.price` and `token.metadata` produce the identical `Token` (CoinGecko's one
  contract endpoint returns price and metadata together; `normalize()` doesn't branch on `cap`).
- `dexscreener`: `{chain, limit, raw}` — `raw` is the **untouched, still possibly multi-chain**
  `/latest/dex/search` response (chain-filtering is `normalize()`'s job, not the HTTP step's, per
  the same "narrowing only inside `normalize()`" note); `limit` defaults to `DEFAULT_LIMIT = 10`
  when the caller's `args.limit` is absent/non-positive.
- `defillama`: `{chain, raw}` — `raw` is the untouched, full multi-chain `/protocol/{slug}`
  response; `normalize()` does the `chainTvls[CHAIN_TVL_KEY[chain]]` slice (§3.2).

**Implementation decisions confirmed by a live probe, not guessed (§11/vendor-drift discipline,
all dated 2026-07-22):**

- CoinGecko's `platform` path segment and `detail_platforms` key both equal the literal `Chain`
  string (`'ethereum'`/`'solana'`) — no translation table needed.
- **DexScreener has no keyless "list newest pairs for chain X" endpoint** — `/token-profiles/
latest/v1` and `/token-boosts/latest/v1` carry neither liquidity/volume data nor a chain filter.
  The confirmed, reliable, keyless endpoint that DOES carry full `Pool`-shaped fields is
  `GET /latest/dex/search?q=<query>`; querying by the target chain's own native asset symbol
  (`NATIVE_QUERY`: `ethereum→'ETH'`, `solana→'SOL'`) reliably surfaces that chain's pairs among the
  (possibly mixed-chain) results, which `normalize()` then filters by `pair.chainId === chain`.
  Documented implementation choice (developer-guidelines §1.6) — the search index isn't
  guaranteed to always return a given chain (e.g. some single-token queries returned zero
  `ethereum` pairs at probe time), but the native-symbol query reliably did across repeated
  probes. **Shape-trap confirmed** (n8n-lesson, explicitly called out in the task file): this
  endpoint returns an OBJECT (`{schemaVersion, pairs: [...]}`), NOT a top-level array — verified
  live and asserted directly in `dexscreener.contract.test.ts` (a different DexScreener endpoint,
  `/tokens/v1/{chainId}/{address}`, returns a top-level array instead — confirmed but NOT used
  here, since it needs a specific token address the `pairs.new`/`pool.info` capability contract
  doesn't provide, only `{chain, limit?}` per ARCHITECTURE.md §5.1).
- DeFiLlama's `chainTvls` object is keyed by **display name** (`"Ethereum"`, `"Solana"`), not the
  lowercase `Chain` value — confirmed against both `/protocol/uniswap` and `/protocol/raydium`
  (`CHAIN_TVL_KEY` map). The protocol-wide `totalTvlUsd` comes from the top-level `tvl` series'
  last point (distinct from the chain-scoped `chainTvls[key].tvl` series used for `tvlUsd`).

`defillama`'s `normalize()` return shape (`ProtocolTvlResult`, exported from this module) is a
**plain TypeScript interface, not a zod schema** — ARCHITECTURE.md §3.2/§5.1 specifies this exact
`{protocol, chain, tvlUsd, totalTvlUsd, source, fetchedAt}` shape literally, not a reference to one
of the six canonical `types/*` schemas; introducing a new zod schema for it would be an
unrequested architectural addition (developer-guidelines §1.6) not asked for by this task.

**`fetch(cap, args)`/`normalize(cap, raw)` are written as object-literal properties with arrow
functions** (`fetch: async (cap, args) => {...}`), not ES2015 method shorthand
(`async fetch(cap, args) {...}`) — a deliberate style choice: method-shorthand syntax puts a bare
`fetch(` token in the source (preceded by whitespace, not a `.`), which is exactly the pattern the
task's "no bare `fetch(`" acceptance grep flags, even though it's a legitimate `ProviderAdapter`
interface-method declaration, not a raw network call. The arrow-function-property form
(`fetch:` — a colon, not an open-paren) is behaviourally identical (both satisfy the interface
structurally) and simply never produces that substring. Doc comments in these three files
similarly avoid ever writing `fetch()`/`fetch(` verbatim — described in prose as "the HTTP step"
instead (task instruction).

### scripts/record-fixture.mjs (new, task 003-4, R-22; extended task 003-5)

Manual dev CLI (plain Node ESM, **not** part of CI — nothing in `.github/workflows/ci.yml`
invokes it, and no test imports it, both verified by this task's own acceptance greps), imported
from the **built** `dist/index.js` (mirrors `packages/mcp-server/scripts/smoke-dist.mjs`'s own
"operate on `dist/`, fail fast if missing" precedent; requires
`pnpm --filter @onchain-intel/core build` first). Usage:
`node packages/core/scripts/record-fixture.mjs <coingecko|dexscreener|defillama|rpc-evm|
rpc-solana|platform-explorer> <chain> [address|protocolSlug]`.

For `coingecko`/`dexscreener`/`defillama`/`rpc-evm`/`rpc-solana`: performs exactly ONE real live
HTTP call per invocation, through the target adapter's own real HTTP step. Writes the call's raw
result (whatever shape that adapter's own fetch step returns — its private `{chain, [limit,] raw}`
envelope, not just the vendor wire body) to `test/fixtures/<adapter>/<name>.json`, plus a sibling
`<name>.evidence.md` recording `recorded_at` (ISO timestamp), `endpoint` (the exact URL touched),
`http_status`, `capability`, `args`, and two field-list lines: `envelope_fields` (this adapter's
own hand-off shape's keys) and `vendor_response_fields` (the actual **vendor** response body's own
top-level keys — the field-list evidence R-22 cares about, for vendor-drift monitoring). The
endpoint/status are captured via an **instrumented `fetchImpl`** injected into the adapter factory
(wraps the global HTTP client, records `{url, status}` as a side effect of the SAME one call) —
never a second, separate probe request. On a live-call failure, no fixture JSON is written; the
evidence file instead records the error and an explicit note that a hand-authored minimal fixture
must be created and clearly marked as such before the corresponding contract test can pass (all
ten live calls across both tasks succeeded — see the fixtures sections above — so no
hand-authored substitute was needed for any of them, except `dash-platform`, which this script
never records at all — see its own section above).

**`platform-explorer` is a deliberate exception to "one invocation, one call"** (task 003-5): ONE
invocation performs FOUR real live calls, one per distinct underlying REST endpoint this adapter's
seven capabilities collapse onto (`/status` alone serves four of them, so it's called once, not
four times) — matching the task file's own single documented command line
(`record-fixture.mjs platform-explorer dash # live + history эндпоинты`), which intentionally asks
for both current-state and history endpoints together rather than four separate invocations. Each
of the four calls writes its own `test/fixtures/platform-explorer/<name>.{json,evidence.md}` pair,
in the exact same evidence format as the single-call path (via a shared `recordOneCall()` helper —
extracted so the two paths share evidence-writing logic instead of duplicating it); a failure in
one of the four does not abort the other three (each is attempted, `process.exitCode` is set to 1
if any failed).
