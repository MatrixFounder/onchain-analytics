### Почему это может быть очень значимым для цены DASH

Раньше Dash критиковали за то, что его приватность основана на CoinJoin, который:

* не обеспечивает криптографическую приватность;
* может анализироваться;
* считается устаревшим по сравнению с zk-SNARKs.

Теперь Dash получил:

* ✅ Halo 2 zk-SNARKs;
* ✅ Orchard Shielded Pool;
* ✅ скрытие отправителя, получателя и суммы;
* ✅ возможность selective disclosure (view keys), что важно для соответствия требованиям регуляторов. ([Dash][2])

Фактически Dash стал первым крупным проектом, который **не разработал собственную систему с нуля, а интегрировал уже проверенную технологию Zcash**.

# 2. Почему ZEC вырос в несколько раз?

Если посмотреть на ончейн, видно несколько очень важных процессов.

## Рост shielded pool

Самая важная метрика Zcash — **не TPS и не количество адресов**.

Главная метрика — сколько ZEC находится внутри приватного пула.

Сейчас там уже около **4,4–5 млн ZEC**, что составляет примерно **30% обращения**, тогда как в начале 2024 года этот показатель был около 8%. ([crypto.news][2])

Для privacy-монеты это аналог TVL у DeFi.

Чем больше монет находится внутри shielded pool:

* выше анонимность;
* выше полезность сети;
* выше ценность самой технологии.

---

## Денежная премия

Blockworks очень точно описывает это понятие.

У Bitcoin денежная премия строится вокруг дефицита.

У Ethereum — вокруг экосистемы.

У Zcash постепенно начала формироваться денежная премия вокруг **невозможности отслеживания капитала**. ([X (formerly Twitter)][3])

Именно это рынок сейчас начал оценивать.

---

## Эффект сети

Shielded pool обладает сильным сетевым эффектом.

Если приватных пользователей становится в два раза больше, приватность возрастает гораздо сильнее, чем в два раза.

Поэтому большие privacy-пулы становятся все более привлекательными.

# 1. Рост Shielded Pool ⭐⭐⭐⭐⭐ (самая важная метрика)

Это аналог TVL в DeFi.

Нужно отслеживать:

* сколько DASH находится внутри Orchard;
* ежедневный приток;
* ежедневный отток;
* чистый прирост.

Самое приятное — эта метрика **заложена в самом протоколе**.

В Platform v4 появились специальные DAPI-эндпоинты:

* `getShieldedPoolState`
* `getShieldedNotesCount`
* `getMostRecentShieldedAnchor`
* `getShieldedAnchors` ([docs.dash.org][1])

Самая интересная команда:

```
getShieldedPoolState
```

Она должна возвращать общий баланс, находящийся в Shielded Pool. Документация уже описывает этот API, хотя на момент публикации отмечает, что публичные узлы еще не везде предоставляют эти эндпоинты. ([docs.dash.org][1])

# 1. Shielded Pool

Это самая неоднозначная метрика.

Теоретически можно получить текущее состояние:

```
Shielded Pool = 125 432 DASH
```

Но вопрос в другом.

Можно ли узнать:

```
1 июля      34 000
2 июля      38 000
3 июля      42 000
...
```

Из документации Platform пока не видно, что существует API вида:

```
getShieldedPoolState(height=...)
```

или

```
getShieldedPoolHistory()
```

То есть **исторический ряд может вообще не храниться в DAPI**.

В этом случае единственный вариант — самому сохранять ежедневные снимки.

---

# 2. Shielded Transactions

Здесь ситуация лучше.

Каждая операция находится в блокчейне.

Если известны:

* высота блока;
* время;
* тип транзакции,

то историю всегда можно восстановить.

Например:

```
Block 1000

Shield

Shield

Transfer

Unshield
```

↓

```
Shield = 2

Transfer = 1

Unshield = 1
```

То есть историю можно пересчитать задним числом.

Но...

Это может занять часы или даже дни вычислений.

Поэтому почти все аналитические сервисы строят собственую индексирующую базу данных.

---

# 3. Identities

Их можно получить ретроспективно.

Поскольку Identity — это документ Platform.

Если DAPI позволяет делать запросы по времени или по высоте блока,

можно восстановить историю.

Но если DAPI возвращает только текущее состояние,

то историю опять придется строить самостоятельно.

---

# 4. Data Contracts

Здесь проще.

Каждый контракт имеет:

* timestamp;
* tx;
* owner;
* block.

То есть историю можно восстановить полностью.

---

# 5. Documents

То же самое.

Можно получить:

```
Document Created

↓

Document Updated

↓

Document Deleted
```

Но опять же,

если нет собственного индексатора,

придется обходить весь блокчейн.

# Самая большая проблема

Почти все RPC в большинстве блокчейнов отвечают только на вопрос:

> "Что происходит сейчас?"

А инвестору нужен совсем другой вопрос:

> "Как изменилась ситуация за последние 90 дней?"

Это две совершенно разные задачи.

# Как делают Glassnode и CryptoQuant

Они **не делают запросы "за последние 365 дней"**.

Они:

```

↓

Parser

↓

Indexer

↓

PostgreSQL

↓

API

↓

Dashboard
```

То есть они индексируют каждый новый блок по мере его появления.

Именно поэтому их сервисы работают быстро.

## Каких размеров я бы ожидал

Если использовать именно такой подход (не архив блокчейнов, а специализированную аналитическую платформу), то оценки будут примерно такими:

| Сеть      | Полная нормализованная история за месяц | Аналитически важные события за месяц |
| --------- | --------------------------------------: | -----------------------------------: |
| Dash      |                                 ~0.7 ГБ |                        **50–150 МБ** |
| Zcash     |                                 ~0.2 ГБ |                         **30–80 МБ** |
| Bitcoin   |                                   ~5 ГБ |                         **0.5–1 ГБ** |
| Ethereum  |                                 ~100 ГБ |                          **5–20 ГБ** |
| Solana    |                             ~500–700 ГБ |                         **20–80 ГБ** |
| Berachain |                                ~5–15 ГБ |                         **0.5–2 ГБ** |

Поэтому я бы с самого начала проектировал платформу не как «архив блокчейнов», а как **движок для сбора и анализа значимых событий**. Такой подход радикально уменьшает требования к хранилищу, упрощает масштабирование и при этом сохраняет практически всю ценность для инвестора и аналитика.

## Если выбирать одно направление

Если бы я запускал этот проект с нуля, я бы шел поэтапно:

1. Построить универсальный движок сбора и хранения данных.
2. На его основе создать собственные производные метрики и рейтинги — это то, что трудно скопировать.
3. Использовать эти метрики для собственного инвестирования и проверки их ценности на практике.
4. Когда станет понятно, какие сигналы действительно работают, превратить их в продукт: подписку, API или AI-аналитика.

Такой порядок важен, потому что самое ценное в подобном бизнесе — не база данных сама по себе, а накопленная история и проверенные модели, которые позволяют находить полезные закономерности раньше других.

Если сильно упростить, то Dune — это:

У них есть огромная база данных, и пользователь пишет:

# И вот здесь появляется самое большое отличие

На мой взгляд, Dune отвечает на вопрос:

> **"Какие данные есть?"**

Glassnode отвечает:

> **"Какие метрики есть?"**

А платформа, которую мы обсуждаем, может отвечать на вопрос:

> **"Что происходит, почему это происходит и на что стоит обратить внимание?"**

Это уже другой класс продукта.

## И еще одна идея, которая может стать сильным преимуществом

Зная, что ты активно используешь AI-агентов, Claude Code, Obsidian и строишь автоматизированные рабочие процессы, я бы сделал платформу **AI-first**, а не dashboard-first.

То есть дашборды были бы лишь одним из способов просмотра данных. Основной интерфейс — это агент, которому можно задавать исследовательские вопросы:

* «Какие проекты начали демонстрировать фундаментальное ускорение за последний месяц?»
* «Найди сети с ростом активности разработчиков, но без роста цены».
* «Покажи проекты, где пользовательская активность растет быстрее капитализации».

В таком случае собственная историческая база данных становится не только источником графиков, но и базой знаний, на которой AI может строить анализ и находить закономерности. Это существенно отличается от Dune, где основным инструментом взаимодействия остается запрос к данным.
